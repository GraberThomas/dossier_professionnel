\section{Présentation du projet}

FeastVerse est un projet universitaire réalisé dans le cadre d'un enseignement consacré au développement Java Spring Boot. L'objectif du projet était de concevoir une API REST autour d'un univers de recettes de cuisine. L'application devait permettre de gérer plusieurs ressources liées entre elles : les utilisateurs, les ingrédients, les types d'ingrédients, les recettes, les commentaires, les likes et les signalements.

Ce projet est mobilisé dans ce dossier comme preuve principale de développement back-end applicatif. Contrairement au projet professionnel FCU / Ediacara, qui porte surtout sur l'adaptation d'une chaîne existante, FeastVerse permet de présenter une application conçue autour d'une architecture plus classique d'API web : contrôleurs REST, services métier, repositories, entités de persistance, DTO, sécurité, documentation et tests fonctionnels.

L'intérêt du projet ne réside pas uniquement dans la création de routes HTTP. Il repose également sur la mise en place d'une logique métier structurée : authentification par jeton, gestion des rôles, distinction entre ressources publiques et privées, règles de modération, validation des entrées, gestion des erreurs, pagination, tri et filtrage.

\section{Périmètre fonctionnel et limites du projet}

Le périmètre fonctionnel de FeastVerse couvre plusieurs domaines. Les utilisateurs peuvent créer un compte, s'authentifier, consulter des recettes, interagir avec les contenus, commenter, liker et signaler des ressources. Les modérateurs disposent de fonctionnalités liées au traitement des signalements, tandis que les administrateurs disposent de droits plus étendus sur certaines ressources.

Les ressources principales du projet sont les suivantes :

\begin{itemize}
    \item les utilisateurs et leurs rôles ;
    \item les ingrédients et les types d'ingrédients ;
    \item les recettes et les étapes de recettes ;
    \item les commentaires ;
    \item les likes ;
    \item les signalements ;
    \item les types de recettes et certaines ressources de référence.
\end{itemize}

Le projet doit toutefois être présenté avec ses limites. Toutes les fonctionnalités n'étaient pas finalisées au même niveau. Les ressources liées aux utilisateurs, ingrédients, commentaires, likes et signalements sont les plus avancées. En revanche, les recettes et les étapes de recettes étaient principalement disponibles en lecture au moment du rendu initial du projet. La création, la modification et la suppression complètes de ces ressources faisaient partie des perspectives d'évolution.

Cette limite est importante dans l'analyse du projet. Elle montre que FeastVerse n'est pas présenté comme une application terminée en production commerciale, mais comme une API back-end avancée permettant de démontrer des compétences de conception, de développement, de sécurisation, de validation et de documentation.

\section{Architecture technique de l'API}

\subsection{Stack utilisée}

L'application a été développée en Java avec Spring Boot. Le projet s'appuie sur Gradle pour la gestion du build et des dépendances. La stack technique mobilise notamment Spring Web pour l'exposition des endpoints REST, Spring Data JPA pour la persistance, Hibernate comme ORM, Spring Security pour la sécurisation des accès, JWT pour l'authentification, PostgreSQL pour la base de données et MapStruct pour les mappings entre entités et DTO.

L'utilisation de Spring Boot permet de structurer rapidement une API robuste, tout en conservant une organisation claire entre les différentes couches de l'application. Le choix de PostgreSQL est cohérent avec le modèle relationnel du projet, qui repose sur des entités fortement liées : utilisateurs, recettes, ingrédients, commentaires, likes et signalements.

\subsection{Organisation en couches}

L'architecture de FeastVerse suit une organisation en couches classique pour une API REST :

\begin{itemize}
    \item les contrôleurs exposent les endpoints HTTP ;
    \item les services portent la logique métier ;
    \item les repositories assurent l'accès aux données ;
    \item les entités représentent le modèle de persistance ;
    \item les DTO contrôlent les données reçues et renvoyées par l'API ;
    \item les mappers assurent la conversion entre entités et DTO.
\end{itemize}

Cette séparation permet de limiter le couplage entre la représentation interne des données et les réponses exposées aux clients de l'API. Par exemple, un utilisateur administrateur ne reçoit pas nécessairement la même vue d'une ressource qu'un utilisateur standard. Les DTO permettent donc d'adapter la réponse au contexte d'utilisation et au niveau d'autorisation.

Cette organisation m'a également permis d'éviter de placer la logique métier directement dans les contrôleurs. Le rôle du contrôleur est de recevoir la requête, valider les paramètres principaux, déléguer au service approprié, puis transformer le résultat en réponse HTTP. Le service porte la logique métier, tandis que le repository reste centré sur l'accès aux données.

\subsection{Persistance et modèle de données}

La persistance repose sur Spring Data JPA et Hibernate. Les repositories permettent d'interroger la base sans écrire systématiquement de requêtes SQL manuelles. Cette approche facilite la mise en place de recherches paginées, triées et filtrées, tout en conservant une structure de code homogène.

Le modèle métier comporte plusieurs relations importantes : un utilisateur peut créer des ressources, aimer des recettes, commenter des recettes ou des étapes, et créer des signalements. Une recette peut être associée à des ingrédients, des étapes, des commentaires, des likes et des signalements. Cette structure impose de réfléchir à la cohérence des données, à la visibilité des ressources et aux droits d'action selon le rôle de l'utilisateur connecté.

\section{Recherche, pagination et filtrage}

Une partie importante du projet concerne la recherche et la consultation des ressources. L'API ne se limite pas à renvoyer l'ensemble des données : elle permet de filtrer, trier et paginer les résultats. Ce choix est essentiel pour une API qui manipule plusieurs centaines d'ingrédients et plusieurs ressources liées aux utilisateurs.

L'exemple suivant illustre la logique appliquée aux recettes. Le contrôleur accepte plusieurs paramètres optionnels, puis les transmet au service métier. L'objet \texttt{Pageable} permet de gérer la pagination et le tri directement dans le flux de la requête.

\paragraph{Extrait de code : filtrage et pagination des recettes}

\begin{verbatim}
@GetMapping
public Page<RecipeViewDto> getAllRecipes(
        @RequestParam(required = false) String name,
        @RequestParam(required = false) Integer maxTotalTime,
        @RequestParam(required = false) RecipeDifficulty difficulty,
        @RequestParam(required = false) UUID typeId,
        @RequestParam(required = false) List<Long> withIngredient,
        @RequestParam(required = false) List<Long> withIngredientType,
        @RequestParam(required = false) List<Long> withoutIngredientType,
        @RequestParam(required = false) List<String> withTags,
        @RequestParam(required = false) UUID ownerId,
        @RequestParam(required = false, defaultValue = "PUBLIC")
        VisibilityFilter visibility,
        @ParameterObject
        @PageableDefault(size = 10, sort = "likeCount",
                         direction = Sort.Direction.DESC)
        Pageable pageable
) {
    Page<Recipe> recipes = recipeService.findAllRecipes(
            name, maxTotalTime, difficulty, typeId,
            withIngredient, withIngredientType,
            withoutIngredientType, withTags,
            ownerId, visibility, pageable
    );
    return recipes.map(recipeMapper::toRecipeListViewDto);
}
\end{verbatim}

Cet extrait est intéressant car il montre plusieurs choix de conception. D'abord, les critères de recherche restent optionnels, ce qui permet de combiner plusieurs usages dans un même endpoint. Ensuite, la pagination est intégrée à la signature de la méthode et ne nécessite pas de format de réponse spécifique. Enfin, la conversion vers un DTO de liste évite de renvoyer l'entité complète et permet de limiter la réponse aux informations utiles pour une page de résultats.

\section{Sécurité et gestion des rôles}

\subsection{Authentification par JWT}

La sécurité est l'un des points importants du projet. FeastVerse utilise une authentification par JWT. Après authentification, l'utilisateur reçoit un jeton qui doit être transmis avec les requêtes protégées. Un filtre dédié intercepte les requêtes, extrait le jeton, vérifie sa validité et initialise le contexte de sécurité Spring.

Cette approche permet de rendre l'API sans état côté serveur. Le serveur n'a pas besoin de conserver une session utilisateur classique ; les informations nécessaires à l'identification sont portées par le jeton et validées à chaque requête.

\paragraph{Extrait de code : extraction et validation du JWT}

\begin{verbatim}
final String authorizationHeader = request.getHeader("Authorization");

String jwt = null;
UUID userId = null;

if (authorizationHeader != null && authorizationHeader.startsWith("Bearer ")) {
    jwt = authorizationHeader.substring(7);
    userId = jwtUtil.extractUserId(jwt);
}

if (userId != null &&
    SecurityContextHolder.getContext().getAuthentication() == null) {

    UserDetails userDetails = userDetailsService.loadUserById(userId);

    if (jwtUtil.validateToken(jwt, userId)) {
        UsernamePasswordAuthenticationToken authToken =
                new UsernamePasswordAuthenticationToken(
                        userDetails,
                        null,
                        userDetails.getAuthorities()
                );
        SecurityContextHolder.getContext().setAuthentication(authToken);
    }
}
\end{verbatim}

Cet extrait met en évidence le fonctionnement du filtre de sécurité. La requête est analysée avant d'atteindre les contrôleurs. Si un jeton est présent et valide, l'utilisateur est chargé puis placé dans le contexte de sécurité. Les contrôleurs peuvent ensuite s'appuyer sur ce contexte pour appliquer les règles d'autorisation.

\subsection{Rôles et autorisations}

Trois rôles principaux sont utilisés dans l'application : utilisateur standard, modérateur et administrateur. Ces rôles ne donnent pas accès aux mêmes fonctionnalités. Un utilisateur standard peut par exemple créer certaines ressources, commenter, liker ou signaler un contenu. Un modérateur peut accéder aux signalements et traiter leur statut. Un administrateur dispose de droits plus étendus, notamment pour certaines suppressions définitives.

La protection des endpoints repose sur les annotations de sécurité de Spring. Cette logique rend les règles d'accès lisibles directement au niveau des contrôleurs. Elle permet également d'exprimer des règles plus fines, par exemple autoriser un utilisateur à modifier son propre profil tout en réservant certaines actions à l'administrateur.

\paragraph{Extrait de code : protection d'endpoints par rôle}

\begin{verbatim}
@PreAuthorize("hasRole('ROLE_STANDARD')")
@PostMapping(consumes = MediaType.MULTIPART_FORM_DATA_VALUE)
public IngredientViewDto createIngredient(...) { ... }

@PreAuthorize("hasAnyRole('ROLE_MODERATOR', 'ROLE_ADMINISTRATOR')")
@GetMapping("/reports")
public Page<ReportViewDTO> getAll(...) { ... }

@PreAuthorize("hasRole('ROLE_ADMINISTRATOR')")
@DeleteMapping("/reports/{reportId}")
public ResponseEntity<Void> deleteReport(...) { ... }
\end{verbatim}

Cet extrait montre que les droits ne sont pas seulement déclarés dans une documentation externe. Ils sont intégrés dans le code, au niveau des endpoints concernés. Cela facilite la lecture des règles de sécurité et limite le risque d'oublier un contrôle dans la logique métier.

\section{Fonctionnalités métier développées}

\subsection{Gestion des utilisateurs}

La partie utilisateur permet de gérer l'inscription, l'authentification, la consultation de profils et la modification de certaines informations. L'API distingue plusieurs niveaux de détail dans les réponses : une vue publique, une vue personnelle et une vue administrateur. Cette distinction permet d'éviter d'exposer les mêmes informations à tous les clients.

La liste des utilisateurs peut être paginée, triée et filtrée selon plusieurs critères, par exemple le rôle, le nom, le prénom, le pseudo ou l'adresse électronique. Cette fonctionnalité montre l'utilisation de filtres dynamiques et de réponses paginées adaptées à une API REST.

\subsection{Gestion des ingrédients}

La gestion des ingrédients constitue une partie importante du projet. L'API permet de consulter les types d'ingrédients, de rechercher des ingrédients, de filtrer les résultats, de créer de nouveaux ingrédients et d'associer une image à une ressource à l'aide d'une requête multipart.

Cette partie illustre plusieurs problématiques back-end : validation des données d'entrée, upload de fichier, distinction entre ressource publique et privée, propriété d'une ressource, suppression logique et suppression définitive. La suppression définitive est réservée à l'administrateur, ce qui permet de montrer une règle d'autorisation métier plus fine qu'un simple accès authentifié.

Le système de signalement permet aux utilisateurs de signaler des contenus ou comportements problématiques. Les modérateurs et administrateurs peuvent ensuite consulter les signalements et mettre à jour leur état. L'administrateur dispose en plus de droits de suppression.

Cette fonctionnalité est intéressante dans le cadre du référentiel, car elle montre la prise en compte de règles métier différenciées selon les rôles. Elle permet aussi de démontrer une logique d'autorisation plus réaliste qu'une application où tous les utilisateurs disposent des mêmes droits.

\section{Qualité des réponses et gestion des erreurs}

Une API doit être prévisible pour ses clients. FeastVerse intègre donc une gestion centralisée des erreurs afin de produire des réponses structurées en cas de problème. Les erreurs de validation, les formats invalides, les ressources inexistantes ou les conflits métier peuvent ainsi être transformés en réponses HTTP cohérentes.

Cette gestion des erreurs est importante pour l'expérience développeur. Elle évite de renvoyer des traces techniques brutes et permet au client de comprendre plus facilement la cause d'une erreur. Elle participe également à la qualité générale de l'API, car les cas d'échec sont traités comme des comportements attendus et non comme des situations imprévues.

\paragraph{Extrait de code : réponse d'erreur de validation}

\begin{verbatim}
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<ErrorResponse> handleValidationExceptions(
        MethodArgumentNotValidException ex,
        WebRequest request
) {
    Map<String, String> errors = new HashMap<>();

    for (FieldError error : ex.getBindingResult().getFieldErrors()) {
        errors.put(error.getField(), error.getDefaultMessage());
    }

    ErrorResponse response = new ErrorResponse(
            HttpStatus.BAD_REQUEST.value(),
            HttpStatus.BAD_REQUEST.getReasonPhrase(),
            "Validation failed",
            getPath(request),
            errors
    );

    return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(response);
}
\end{verbatim}

Cet extrait montre que les erreurs de validation ne sont pas laissées au comportement par défaut du framework. Elles sont transformées en réponse structurée, avec un statut HTTP, un message lisible, le chemin concerné et le détail des champs invalides.

    pm.expect(pm.response.code).to.equal(200);

    pm.expect(jsonData).to.have.property("accessToken").that.is.not.empty;

\section{Limites et bilan critique}

Le projet présente plusieurs limites. La première concerne le périmètre fonctionnel : certaines ressources, notamment les recettes et les étapes de recettes, n'étaient pas encore entièrement disponibles en écriture au moment du rendu initial. La deuxième concerne les tests : la collection Postman permet de valider de nombreux scénarios fonctionnels, mais elle ne remplace pas une couverture complète par tests unitaires et tests d'intégration automatisés. Enfin, la documentation doit rester synchronisée avec l'état réel du projet, notamment lorsque le mode de déploiement évolue.

Malgré ces limites, FeastVerse m'a permis de consolider des compétences importantes en développement back-end. Le projet m'a amené à concevoir une API REST structurée, à mettre en place une authentification JWT, à gérer des rôles et des autorisations, à organiser le code en couches, à manipuler une base PostgreSQL, à documenter les endpoints et à valider fonctionnellement les principaux scénarios avec Postman.

Ce projet constitue donc une preuve pertinente pour démontrer ma capacité à développer une solution back-end complète, sécurisée, testable et documentée, tout en prenant du recul sur ses limites et ses perspectives d'amélioration.
